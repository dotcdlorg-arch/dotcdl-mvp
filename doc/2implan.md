## 1. P1 #9 — History progress loading on mount

**Goal:** When a signed-in user opens `/practice` or `/signs`, hydrate the local `progress` state from `GET /api/progress` so the sidebar stats and per-question status badges reflect their actual history instead of starting from zero.

**Files to modify:**
- [`app/practice/page.js`](../app/practice/page.js) — add a mount-time fetch + state hydration (will be replaced by `useProgress` hook in #14)
- [`app/signs/page.js`](../app/signs/page.js) — same
- No backend change — `GET /api/progress` already exists at [`app/api/progress/route.js`](../app/api/progress/route.js) and returns `{ questionProgress, signProgress, recentMocks }`.

**Step-by-step plan:**
1. In `PracticeInner` (practice page), add a `useEffect(() => { ... }, [])` that does `fetch('/api/progress')` with `credentials: 'same-origin'` (Clerk cookies). Verify: in DevTools Network tab, one GET fires on first render.
2. Shape-transform the response: server returns an array `questionProgress: [{ question_code, status, last_score, last_transcript, updated_at, ... }]`. Map it into the keyed object shape the local state already uses: `{ [code]: { status, score, transcript, viewCount: 1 } }`. Verify: console.log shape matches what `markStatus` writes.
3. On 401 (unauthenticated), silently no-op (don't break the page for anonymous flow). On 5xx, swallow and `console.warn` only. Verify: log out → open page → no error, no toast yet (toasts arrive in item #3).
4. After successful hydration, the existing `stats` derivation block in practice (line 372–381) automatically picks up real `seen / understood / review / avgScore`. Verify visually: sidebar numbers > 0 for a user who has answered before.
5. Same change in `app/signs/page.js`: read `signProgress: [{ sign_code, score, ... }]`, shape into the `{ [sign_code]: score }` map the page already uses (line 79).
6. Add a small `loading` flag (boolean) so on first render we don't flash "0 seen / 0 understood" when we know we're about to fetch. Skip rendering stats until the fetch resolves OR after 800ms (whichever first) to avoid layout shift. Verify: hard refresh on `/practice` shows no zero-flash on a fast network.

**Risks / pitfalls:**
- **Race with `markStatus`:** if the user clicks "Understood" within the first ~200ms while the GET is in flight, the POST optimistic update could be clobbered by the late-arriving GET. Mitigation: only hydrate state for keys NOT already present in local `progress` (i.e., merge with local taking priority).
- **Optimistic UI regression:** keep `markStatus` writing local state immediately (the existing pattern). The mount-time GET only fills *initial* state; subsequent writes stay optimistic.
- **Anonymous users:** `/practice` should still work logged out. Treat 401 as "no history" silently.
- **Stats display change:** sidebar stats currently include `mocks` count derivation that this page doesn't compute. Item #9 covers only question/sign progress, not mocks count.

**Verification criteria:**
- Sign in → answer 3 questions in `/practice` → mark 2 understood → reload page → sidebar shows seen=3, understood=2. Currently shows 0/0.
- Same flow in `/signs`.
- Network tab shows exactly one GET `/api/progress` per mount.
- Logged-out user can still see questions render normally.

**Estimated LoC:** +60 lines (≈ 30 per page including merge logic), 0 removed. ½–1 day.

**Step-by-step plan:**

3. **SSR hydration trap:** Next.js renders server-side first where `localStorage` is undefined → server uses `'zh'`, client may differ → hydration mismatch warning. - **Hydration mismatch** — handled by initializing server-side with default, then hydrating client-side from localStorage in a `useEffect`. This is a well-known Next.js pattern; document it in a comment so future devs don't "fix" it back.

**Verification criteria:**

- Cross-page nav doesn't reset (practice → signs → drive → terms all share).

**Files to modify:**

**Proposed signature:**
```js
const { isRecording, error, audioBlob, startRecording, stopRecording, reset } = useRecorder({
  onStop: async (blob) => { /* caller-provided: transcribe+score */ },
  mimeType: 'auto', // resolved via MediaRecorder.isTypeSupported
})
```

**Step-by-step plan:**
1. Build `useRecorder.js`. Internal state: `isRecording`, `error`, `audioBlob`. Refs: `mrRef`, `streamRef`. On mount: nothing. On `startRecording()`: synchronously call `unlockAudio()` BEFORE `await getUserMedia` (this is the iOS gesture-preservation requirement — Actions 60–61). On `mr.onstop`: stop tracks, build blob, set `audioBlob`, call `onStop(blob)` if provided. Verify: console.log the mimeType chosen.
2. Move `unlockAudio()` from `practice/page.js` into `lib/audio-unlock.js`. Keep the persistent `<audio>` element + silent WAV pattern exactly. Practice page imports it back and uses it for autoplay too. Verify: `/practice` continuous play on iPhone Safari still plays audio (regression check from Actions 60–61).
3. Migrate `practice` first. Wire `onStop` to the existing `/api/transcribe` POST → set `transcript`. Verify: record + transcribe still works.
4. Migrate `mock`. Note that mock's `onStop` does more (transcribe → score → state update); pass an `onStop` that handles all of it. Verify: speak mode in `/mock` still scores correctly.
5. Migrate `drive`. Its `onStop` triggers `scoreAndAdvance(text)`. Verify: officer/driver bubbles still appear, scores save.

**Files to create:**

**Files to modify:**

**Proposed signature:**

```js
// progress: { [code]: { status, score, transcript } }
// markQuestion(code, status, score, transcript): optimistic local + POST
```

**Step-by-step plan:**
1. Build the hook. On mount: `GET /api/progress` → shape into `{ [code]: {…} }`. State: `progress`, `loading`. Merge logic protects optimistic writes (per item #1 risk note).
2. `markQuestion` writes local state immediately, then POSTs in the background. On POST failure → toast 'warn' (item #3). **Optimistic semantics must be preserved** — UI never waits for the server.

5. Migrate signs. Verify: scoring a sign updates sidebar count.
6. Drive: use only `markQuestion` (no display of progress on this page).

**Risks / pitfalls:**
- **Optimistic UI regression** — CRITICAL. Local `setProgress` must happen synchronously, BEFORE the fetch. If a refactor accidentally `await`s the POST before setting state, the UI lags. Test by throttling network to "Slow 3G" in DevTools.
- **`/signs` uses a different shape:** today its progress map is `{ [sign_code]: scoreNumber }`, whereas practice uses `{ [code]: { status, score, ... } }`. Either normalize both to the object shape, or let `useProgress({ type })` return the right shape per type. Recommend normalizing to the object shape (small migration; ~5 LoC change in signs's render).
- **Bundle with #1:** sequencing-wise, #14 makes #1 cleaner because the mount-time GET lives in one place from day one. Implement #14 first, then #1 is just "wire `progress` into UI."

**Verification criteria:**
- Reload after answering → progress visible on `/practice` and `/signs` (this is item #1).
- Marking understood updates UI in <50ms (optimistic).

# 9. Cross-cutting recommendations

## 9.1 PR sequencing

- **PR 2 → ITEM #8.** `LanguageProvider` + 6 page migrations. Small surface but touches many files; isolate.

- **PR 4a–4e → ITEM #16 in 5 sub-PRs.** One per page (practice, mock, drive, terms, AppShell). Each ~60 LoC moved + smoke test.
- **PR 5 → ITEM #12.** `useRecorder` + `audio-unlock.js`. Critical iOS testing.
- **PR 6 → ITEM #13.** `useScoring` with `scoreKeywords` fallback preserved.
- **PRs 7.1–7.5 → ITEM #15.** Phase A, B, C-QuestionCard, C-drive, D (optional).

**Bundling recommendations summarized:**
- **Bundle #14 + #9** (saves ~½ day, avoids duplicate scaffolding).
- **Do NOT bundle #12 + #13** even though they touch the same pages — the iOS regression risk in #12 deserves dedicated verification before scoring logic moves around too. Sequential, not bundled.
- **Do NOT bundle #16 (i18n) with #15 (components)** — would create a 1000-line diff. #16 must ship first so #15's components can call `t(lang, key)` directly.

## 9.2 Verification gates between PRs

After each PR before merging:
1. `npx next build` clean (no new warnings).
2. Manual smoke test: sign in → `/practice` → answer one → mark understood → reload → still understood.
3. Mobile gesture test (after #5 + #12): on iPhone Safari, continuous play still produces audio, recording still transcribes.
4. Language test: switch to Spanish on landing → navigate to each page → still Spanish (after #8); each page renders without missing-key warnings (after #16).
5. Fallback test (after #13): block `/api/score` in DevTools → score still appears from `scoreKeywords`.
6. Toast non-spam check (after #10): trigger 3 mic errors in 1 second → 1 toast (deduped).

## 9.3 Rollback strategy

- Each PR is independent and individually revertable via `git revert <sha>`. No PR depends on a database migration; no PR removes a public API.
- The `useRecorder` extraction (#12) is the highest-risk single PR. Keep the old inline `MediaRecorder` blocks in git history; if iOS playback regresses post-merge, revert PR 5 cleanly without touching subsequent PRs (they don't depend on the hook's internal shape, only on the recorder API surface).
- The component-library PRs (#15) are visual only; rolling back any one phase doesn't affect data or auth.
- For #16 (i18n): if a key migration breaks one language, revert just that page's sub-PR. Other languages and other pages are unaffected because each page's migration is its own commit.

## 9.4 Behavioral checks for CLAUDE.md compliance

Each PR must satisfy:
- **Surgical changes:** every changed line traces to its item's goal. Don't "improve" adjacent inline styles, don't refactor unrelated comments.
- **Simplicity first:** if any item's implementation grows past the LoC estimate by 50%, stop and review — likely overengineering (e.g., don't add config flags to `useRecorder` for "future flexibility"; ship the minimum).
- **Goal-driven execution:** verification criteria above translate directly into manual or automated checks before each merge.
